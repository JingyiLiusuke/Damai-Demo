# Android 混合状态机抢票助手设计

## 1. 目标

将 `damaiv2.py` 的三阶段操作迁移为仅供个人侧载的 Android 原生 APK，去除
PC、Appium HTTP、UiAutomator2 Server 和 USB 命令往返对核心操作路径的影响。

第一版采用：

1. 无障碍事件驱动状态机作为页面状态主检测路径。
2. `dispatchGesture()` 坐标点击作为阶段一的最快路径，以及后续阶段的兜底路径。
3. 无障碍节点 ID/文字查找作为阶段二、阶段三的主点击路径。
4. `AccessibilityService.takeScreenshot()` 局部模板匹配作为节点和坐标均不可靠时的最后兜底。

APK 不发布到应用商店，不尝试绕过验证码、风控、登录验证或服务端排队。

## 2. 范围

### 包含

- 配置目标时间、提前触发偏移和三个阶段的点击区域。
- 手动启动、停止和查看运行状态。
- 仅监听 `cn.damai` 的无障碍事件。
- 执行“立即预定 -> 确定票价 -> 立即提交”三阶段状态机。
- 记录每次事件、检测、点击和状态迁移的单调时钟时间。
- 提供调试采集功能，保存无障碍节点摘要和按钮区域截图。
- 对点击次数、阶段时长和截图频率设置硬限制。

### 不包含

- 自动登录、验证码识别、实名信息填写和支付。
- 抓取或调用大麦内部网络接口。
- Root、Shizuku 或系统级输入注入。
- 多设备控制、云端配置和持续后台常驻。
- 第一版中的连续 `MediaProjection` 录屏分析。

## 3. 工程与运行条件

- Kotlin 原生 Android 应用。
- 最低系统版本为 Android 11（API 30），目标测试设备为 Android 12。
- 使用 JDK 17 和现代 Android Gradle Plugin。
- 应用包名使用 `com.local.damaiassistant`。
- 用户手动侧载 APK、开启无障碍服务并授予截图能力。
- 运行前由用户完成登录、选择场次，并将大麦停留在项目详情页。
- 运行期间手机保持亮屏、解锁，大麦保持前台。

当前本机只有 Android SDK 29 和旧 JDK 8。实现前需要安装 JDK 17，并补充至少
API 30 的 Android SDK；建议使用 Android Studio 管理构建工具。

## 4. 组件

### 4.1 `MainActivity`

负责配置和运行控制，不直接执行自动化操作：

- 目标时间和提前触发毫秒数。
- 阶段一、二、三的按钮矩形。
- 每阶段最长时间、最大点击次数和重试间隔。
- 视觉模板、匹配阈值和视觉兜底开关。
- 无障碍服务状态、当前状态、最近日志。
- “校准”“开始待命”“立即停止”操作。

所有按钮矩形保存为屏幕宽高比例。导入现有像素坐标时，先使用当前显示尺寸转换，
避免分辨率变化后直接复用错误坐标。

### 4.2 `DamaiAccessibilityService`

仅接收大麦包名的以下事件：

- `TYPE_WINDOW_STATE_CHANGED`
- `TYPE_WINDOW_CONTENT_CHANGED`
- `TYPE_VIEW_SCROLLED`

职责：

- 将原始事件转交状态机，不在回调中执行昂贵操作。
- 提供按 view ID、文字和 content description 查找节点的能力。
- 对不可点击节点向上查找首个可点击父节点。
- 通过 `dispatchGesture()` 点击坐标。
- 按需调用 `takeScreenshot()`。
- 在停止、超时、切换到非大麦应用时取消未完成任务。

### 4.3 `TicketStateMachine`

状态定义：

```text
IDLE
ARMED
STAGE_1_RESERVE
STAGE_2_CONFIRM_PRICE
STAGE_3_SUBMIT
DONE_PENDING_RESULT
DONE
FAILED
CANCELLED
```

状态机运行在单独的 `HandlerThread` 上。无障碍事件、定时触发、手势回调和截图结果
全部串行投递到该线程，防止重复点击和状态竞争。

### 4.4 `NodeDetector`

检测顺序：

1. 精确 view ID。
2. 精确文字。
3. `contentDescription`。
4. 已知按钮矩形内的可点击节点。

每次只读取当前状态所需的少量节点，不遍历并序列化完整窗口树。

### 4.5 `VisualFallback`

仅在当前阶段的节点检测和坐标点击未能推进状态时启用：

1. 调用 `takeScreenshot()` 获取最新画面。
2. 按当前阶段配置裁剪按钮局部区域。
3. 转为缩小后的灰度图。
4. 使用归一化模板差异进行匹配。
5. 分数超过阈值时点击匹配中心。

第一版不引入 OpenCV，避免扩大 APK 和增加原生库兼容成本。模板和目标区域尺寸统一
后使用 Kotlin 实现局部像素比较。截图之间至少间隔 400 毫秒，并限制每阶段截图次数。

### 4.6 `RunLogger`

使用环形内存日志并可导出文本文件，记录：

- 墙上时钟和 `SystemClock.elapsedRealtimeNanos()`。
- 状态迁移原因。
- 无障碍事件类型。
- 节点检测耗时、截图匹配耗时。
- 手势发起和完成时间。
- 点击后首次观察到下一阶段的时间。

日志不得记录账号、身份证、手机号或订单详情。

## 5. 状态流

### 5.1 待命

用户点击“开始待命”后进行前置检查：

- 无障碍服务已开启。
- 当前前台包名为 `cn.damai`。
- 目标时间尚未过期。
- 三个阶段的坐标区域有效。
- 手机屏幕已开启。

检查通过后进入 `ARMED`。目标时间距离较远时低频等待；最后两秒使用
`SystemClock.elapsedRealtimeNanos()` 根据墙上时钟换算出的单调时钟截止点进行等待。
第一版要求用户在目标时间前手动待命，不依赖系统精确闹钟唤醒。

### 5.2 阶段一：立即预定

到达“目标时间 - 提前偏移”后：

1. 立即进入 `STAGE_1_RESERVE`。
2. 不等待节点或无障碍事件，直接点击阶段一坐标中心。
3. 等待大麦内容变化事件，并检测 `cn.damai:id/btn_buy_view`。
4. 未推进时按受控间隔重试坐标点击。
5. 发现阶段二节点后立即迁移到阶段二。

不得并发提交多个手势。后续点击只能在上一个手势完成或重试截止时间到达后发起。

### 5.3 阶段二：确定票价

进入 `STAGE_2_CONFIRM_PRICE` 后：

1. 查找 `cn.damai:id/btn_buy_view`。
2. 对节点或其可点击父节点执行 `ACTION_CLICK`。
3. 节点短时间内不可见时，点击阶段二坐标。
4. 坐标仍未推进时，执行一次局部视觉匹配。
5. 检测到文字“立即提交”或对应阶段三节点后进入阶段三。

### 5.4 阶段三：立即提交

进入 `STAGE_3_SUBMIT` 后：

1. 按 view ID、精确文字“立即提交”和 content description 查找节点。
2. 找到后执行 `ACTION_CLICK`。
3. 找不到时点击阶段三坐标。
4. 页面未变化时执行局部视觉匹配并点击匹配中心。
5. 按钮消失、窗口发生预期变化或达到受控提交次数后进入
   `DONE_PENDING_RESULT`。

第一版不把“已发送点击”直接视为订单成功。只有检测到配置的结果页特征时进入
`DONE`；否则保留 `DONE_PENDING_RESULT`，由用户检查页面结果。

## 6. 调度与防重复

- 所有动作使用状态版本号；过期回调不得修改新状态。
- 相同状态下设置点击冷却时间。
- 每阶段有独立的最长时长和最大点击次数。
- 视觉检测执行期间不启动第二次截图。
- 离开 `cn.damai`、用户停止或服务断开时立即取消流程。
- 最终提交阶段默认采用较小点击上限，避免页面已经变化后继续误触支付区域。

具体重试参数不写死为业务常量。第一版提供保守默认值，再根据真机日志调整。

## 7. 校准与调试

真机适配只要求 `adb` 连接，不要求 Appium：

1. USB 开启调试，`adb devices` 能识别设备。
2. 手机解锁并手动打开目标页面。
3. 使用 APK 的调试采集功能记录当前窗口节点摘要和截图。
4. 分别采集三个阶段的按钮矩形和视觉模板。
5. 使用非开售时间或不会产生订单的页面验证状态迁移与点击区域。

Appium Inspector 可以作为辅助工具，用于核对现有脚本中的 view ID、文字和节点层级；
它不进入 APK 的生产运行路径。调试时不能让 Appium 自动化会话与 APK 同时执行点击，
否则测得的延迟和状态会失真。

## 8. 错误处理

- 截图失败：记录原因，本次退回节点/坐标路径，不连续重试截图。
- 节点失效：重新从活动窗口获取根节点，不持有跨页面节点引用。
- 手势被取消：在点击上限内按当前状态重试。
- 服务被系统关闭：流程转为 `FAILED`，重新开启服务后必须由用户重新待命。
- 目标时间已过：默认拒绝待命，用户可显式选择“立即测试”模式。
- 页面未知：停止点击并进入 `FAILED`，避免在非预期页面盲点。

## 9. 测试策略

### 单元测试

- 全部合法和非法状态迁移。
- 点击冷却、超时、点击上限和过期回调。
- 像素坐标与归一化坐标转换。
- 局部模板匹配分数和阈值。
- 墙上时钟到单调时钟截止点换算。

### 仪器测试

- 使用测试 Activity 构造三个阶段页面。
- 验证 ID 节点点击、文字节点点击、父节点点击和坐标兜底。
- 验证事件快速重复时只触发一次动作。
- 验证截图匹配成功、失败和截图限频。

### 真机验证

- 分别测量阶段一直接手势、节点查找点击和视觉兜底耗时。
- 验证不同导航栏状态和屏幕方向不会改变有效坐标。
- Appium/ADB 仅用于采集和旁路观察，不作为正式运行依赖。
- 首次真实操作前先使用不会产生订单的页面完成全流程演练。

## 10. 验收标准

- APK 可在 Android 12 设备侧载并启用无障碍服务。
- 无 PC、Appium Server 或持续 USB 连接时可以独立运行。
- 三阶段均可通过测试页面完成状态迁移。
- 阶段一在目标时刻直接发起坐标手势，不等待节点查询。
- 阶段二、三优先使用节点，节点不可用时按顺序采用坐标和局部视觉兜底。
- 日志可以区分检测耗时、点击耗时和页面响应耗时。
- 任何阶段超过时间或次数上限后停止点击。
- 用户停止、离开大麦或进入未知页面时不会继续自动点击。
